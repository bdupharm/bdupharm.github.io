Casting in PostgreSQL: A Shallow Wade Through Comparisons

Apr 21, 2016

Ever wondered how postgres is performing a comparison in a query like this:

oid is not the same type as regclass… so how is postgres handling this comparison (and others)?

First, we want to query the pg_type table to find the oids of both the oid and regclass types. We need their oids to locate the relevant records on the pg_cast table.

Next, we want to query the pg_cast table.

Now we see that there is implicit casting (castcontext value of i) between the types going both ways.

The castfunc value of 0 means that this cast method does not require a function. Otherwise, we would query the pg_proc table with the oid, value of castfunc, to figure out which function postgres is using.

Lastly, the castmethod is b, which means that the types are “binary-coercible”. I am not sure if this is the correct interpretation, but it may mean that the raw bytes of the two types stay the same between castings.

If you’re ever curious about comparisons in postgres again, you can always poke around in the pg_type, pg_cast and pg_proc tables to figure out what is happening behind the scenes.
